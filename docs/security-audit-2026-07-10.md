# Grove Security Audit — 2026-07-10

Full-repository security audit of the Grove Android app (Kotlin/Jetpack Compose, Org-mode notes). Findings were generated by a security sweep and then adversarially verified against the source; only findings that survived verification with high confidence are reported as vulnerabilities.

## Scope

- All Kotlin sources under `app/src/main`, `AndroidManifest.xml`, Gradle build config, XML resources, and the macrobenchmark module.
- Exported components (`MainActivity` with SEND/VIEW/deep-link intent filters, `CaptureWidgetReceiver`) and all `grove://` deep-link routes (`Routes.kt`, `GroveApp.kt`, `NoteRef.decode`).
- File I/O layer (`SafFileStore`, `JvmFileStore`) for path traversal.
- Room/DAO queries for SQL injection.
- Share-intake and page-title-fetch paths, PendingIntents, signing and telemetry configuration.

Not found / clean: no SQL injection (all Room queries use bound parameters), no path traversal (`JvmFileStore` guards path separators; SAF is bounded to the persisted tree URI), no command injection, no WebView, no hardcoded secrets, all PendingIntents are `FLAG_IMMUTABLE`, release build is minified and non-debuggable with no cleartext-traffic override. WebDAV/Dropbox sync backends are not yet implemented and were out of scope.

## Vuln 1: Data exposure — private note content uploaded to Sentry via crash screenshots

- **Location:** `app/src/main/AndroidManifest.xml:84-99`
- **Severity:** Medium
- **Category:** `sensitive_data_exposure` / third-party telemetry
- **Confidence:** 8/10 (verified in source)

### Description

The Sentry SDK is configured entirely in the main manifest with a hardcoded DSN and:

- `io.sentry.attach-screenshot = true` (line 94)
- `io.sentry.attach-view-hierarchy = true` (line 96)
- `io.sentry.traces.user-interaction.enable = true` (line 92)
- `io.sentry.traces.sample-rate = 1.0` (line 99)

Verification confirmed this applies to **all builds including release**: there is no build-type manifest overlay, no Sentry gating in `buildTypes`, and no Kotlin-side `SentryAndroid.init` — so there is no `beforeSend` / `beforeScreenshotCapture` hook, no screenshot masking, no consent flow, and no settings toggle. The comment on line 93 itself acknowledges the capture "could contain sensitive/PII data."

Grove's entire UI (editor, read mode, search, capture) displays the plaintext contents of the user's private `.org` notes.

### Impact

Any unhandled exception while a note is on screen uploads a full pixel screenshot **and** a serialized view hierarchy (which includes view text content) of that note to sentry.io, without user consent or notice. User-interaction breadcrumbs and 100% trace sampling add behavioral telemetry. Anyone with access to the Sentry project — or a compromise of it — can read the user's private notes. This directly contradicts the app's file-first, private-notes value proposition.

### Recommendation

1. Remove `io.sentry.attach-screenshot` and `io.sentry.attach-view-hierarchy` (or set them to `false`) — these are almost never safe for a notes app.
2. If crash screenshots are genuinely wanted, gate all Sentry telemetry behind an explicit opt-in setting and register a `beforeScreenshotCapture` / `beforeSend` callback that drops attachments when note-bearing screens are active.
3. Disable or reduce `io.sentry.traces.user-interaction.enable` and lower `traces.sample-rate` in production (the manifest's own comment says to adjust it).
4. Consider restricting Sentry to debug builds via a `src/release/AndroidManifest.xml` overlay (`tools:node="remove"`) or a per-build-type DSN placeholder.

## Informational (below reporting threshold)

### Arbitrary-scheme link launch from note content

- **Location:** `app/src/main/java/com/rrajath/grove/ui/screens/ReadNoteScreen.kt:559-561`
- **Assessed severity:** Low (hardening gap; verification confidence 4/10 — filtered from the vulnerability list)

`openOrgTarget` passes any non-`id:`/`#` link target from `[[target][label]]` org syntax directly to `startActivity(Intent(ACTION_VIEW, target.toUri()))` with no scheme allow-list or confirmation. The rendered link shows only the label, hiding the raw target, and a single tap launches the intent. Since `.org` files are synced from external sources, an attacker with write access to the sync directory could plant a label-masked link to any installed app's exported deep-link handler.

Verification downgraded this because the implicit `ACTION_VIEW` intent can only reach exported activities (no intent redirection to unexported components), requires a user tap, and real harm depends on a vulnerable third-party deep link. Still worth fixing as hardening:

- Allow-list schemes (`http`, `https`, `mailto`, `geo`, `tel`) in `openOrgTarget`; for anything else, show the raw target in a confirmation dialog (the existing long-press menu pattern is a natural surface) or refuse to launch.
